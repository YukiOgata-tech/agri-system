# アグリノート移行とデータインポート分析

更新日: 2026-04-27

## 1. このメモの目的

このメモは、アグリノートのエクスポート仕様を前提にして、本アプリで必要な `過去データ移行` と `データインポート機能` の設計方針を整理するためのものです。

---

## 2. 結論

インポート機能は必須です。新規導入時に過去データが捨てられると、記録サービスとしての価値が大きく下がります。

ただし、アグリノートのエクスポートは単一CSV 1枚ではなく、記録種別や明細種別ごとに複数ファイルへ分割される構造です。したがって、本アプリ側も `CSV 1枚読込` ではなく、`ZIP/複数CSV/複数XLSX をまとめて受けるインポートバッチ` として設計した方がよいです。

---

## 4. アグリノートのエクスポート仕様調査

以下は主にアグリノート公式ヘルプの2024年9月5日〜2024年10月29日更新記事を元に整理しています。

### 4-1. 共通仕様

- ブラウザ版の `出力` から Excel / CSV を出力できる
- `日誌データ` と `設定データ` がある
- CSV は `UTF-8（BOM付き）`
- 写真・添付ファイルは出力されない
- ZIP を解凍すると複数ファイルに分かれている場合がある

参照:

- 共通操作: https://agri-note.zendesk.com/hc/ja/articles/900006807123-%E3%83%96%E3%83%A9%E3%82%A6%E3%82%B6%E7%89%88-%E5%87%BA%E5%8A%9B-%E5%85%B1%E9%80%9A%E6%93%8D%E4%BD%9C

### 4-2. 作業記録の出力

作業記録は 6ファイルに分かれます。

- 一般
- 作業者
- 機械
- 農薬
- 肥料
- 資材

重要なのは、`作業ID` `作付ID` `圃場ID` が各ファイルをまたいで結合キーとして使えることです。これらのIDは画面上では確認できないが、エクスポートでは出る、という構造です。

また、複数圃場に対する1件の作業記録を、`圃場あたり` `10aあたり` に按分した出力が存在します。

参照:

- 作業記録出力: https://agri-note.zendesk.com/hc/ja/articles/4402796583577-%E3%83%96%E3%83%A9%E3%82%A6%E3%82%B6%E7%89%88-%E5%87%BA%E5%8A%9B-%E6%97%A5%E8%AA%8C%E3%83%87%E3%83%BC%E3%82%BF-%E4%BD%9C%E6%A5%AD%E8%A8%98%E9%8C%B2

### 4-3. 生育記録の出力

生育記録は 2ファイルです。

- 生育記録
- 生育イベント記録

この構造は、本アプリでいう `環境記録` と `生育観察/イベント` を分ける設計と相性が良いです。

参照:

- 生育記録出力: https://agri-note.zendesk.com/hc/ja/articles/4402796725145-%E3%83%96%E3%83%A9%E3%82%A6%E3%82%B6%E7%89%88-%E5%87%BA%E5%8A%9B-%E6%97%A5%E8%AA%8C%E3%83%87%E3%83%BC%E3%82%BF-%E7%94%9F%E8%82%B2%E8%A8%98%E9%8C%B2

### 4-4. 収穫記録の出力

収穫記録は 1ファイルです。

特徴:

- 収穫ID
- 作付ID
- 圃場ID
- 品質・規格
- 数量

アグリノート側では収穫記録で品質・規格を持ち、さらに収穫ロット番号も扱います。

参照:

- 収穫記録出力: https://agri-note.zendesk.com/hc/ja/articles/4402796800153-%E3%83%96%E3%83%A9%E3%82%A6%E3%82%B6%E7%89%88-%E5%87%BA%E5%8A%9B-%E6%97%A5%E8%AA%8C%E3%83%87%E3%83%BC%E3%82%BF-%E5%8F%8E%E7%A9%AB%E8%A8%98%E9%8C%B2
- 収穫記録の内容: https://agri-note.zendesk.com/hc/ja/articles/900004387666-%E3%83%96%E3%83%A9%E3%82%A6%E3%82%B6%E7%89%88-%E8%A8%98%E9%8C%B2%E7%B7%A803-%E5%8F%8E%E7%A9%AB%E8%A8%98%E9%8C%B2

### 4-5. 出荷記録

出荷記録は 2ファイルです。

- 出荷先
- 収穫元

また、出荷記録の登録・閲覧は管理者権限に限定されます。アグリノートでは出荷が売上情報と直結するためです。

本アプリでも、将来的に出荷/売上を一般記録と権限制御で分ける設計は検討価値があります。

参照:

- 出荷記録出力: https://agri-note.zendesk.com/hc/ja/articles/4402803074585-%E3%83%96%E3%83%A9%E3%82%A6%E3%82%B6%E7%89%88-%E5%87%BA%E5%8A%9B-%E6%97%A5%E8%AA%8C%E3%83%87%E3%83%BC%E3%82%BF-%E5%87%BA%E8%8D%B7%E8%A8%98%E9%8C%B2
- 出荷記録の内容: https://agri-note.zendesk.com/hc/ja/articles/900005360603-%E3%83%96%E3%83%A9%E3%82%A6%E3%82%B6%E7%89%88-%E8%A8%98%E9%8C%B2%E7%B7%A8-%E5%87%BA%E8%8D%B7%E8%A8%98%E9%8C%B2

### 4-6. 設定データ

設定データも出力可能です。

- 圃場情報: 1ファイル
- 作付情報: 4ファイル

つまり、移行対象は `記録` だけではありません。圃場・作付のマスタ移行も同時に考えないと、記録だけ流し込めません。

参照:

- 圃場情報出力: https://agri-note.zendesk.com/hc/ja/articles/4403886311833-%E3%83%96%E3%83%A9%E3%82%A6%E3%82%B6%E7%89%88-%E5%87%BA%E5%8A%9B-%E8%A8%AD%E5%AE%9A%E3%83%87%E3%83%BC%E3%82%BF-%E5%9C%83%E5%A0%B4%E6%83%85%E5%A0%B1
- 作付情報出力: https://agri-note.zendesk.com/hc/ja/articles/4403895499161-%E3%83%96%E3%83%A9%E3%82%A6%E3%82%B6%E7%89%88-%E5%87%BA%E5%8A%9B-%E8%A8%AD%E5%AE%9A%E3%83%87%E3%83%BC%E3%82%BF-%E4%BD%9C%E4%BB%98%E6%83%85%E5%A0%B1

## 5. アグリノートと本アプリの差分

| 観点 | アグリノート | 本アプリ現状 | 差分 |
| --- | --- | --- | --- |
| 場所の概念 | 圃場中心 | 生産エリア中心 | 近いが名称と階層が違う |
| 作業記録 | 複数圃場1件 + 明細分割出力 | 作業記録あり | インポート時に複数CSV結合が必要 |
| 生育/環境 | 生育記録・イベントあり | 環境記録あり、生育観察UIは未整備 | 一部不足 |
| 収穫 | 品質・規格、ロットあり | ロットあり、品質等級あり | 近い |
| 出荷 | 収穫ロット連携、売上、権限制御 | 収穫参照あり、権限制御なし | 権限不足 |
| 設定データ移行 | 圃場・作付も出力可能 | 取込UIなし | 大きい差 |
| 添付ファイル | 画面では管理、出力では含まれない | 写真/証跡は設計あり | 実装途上 |
| インポート | 公式の汎用インポート機能は確認できず | 未実装 | 本アプリで価値を出せる余地大 |

## 6. 本アプリで必要なインポート機能

### 6-1. 受け口

単一CSVアップロードではなく、次を受けられるようにします。

- ZIP
- CSV複数
- XLSX複数

対象:

- 圃場情報
- 作付情報
- 作業記録
- 生育記録
- 収穫記録
- 出荷記録

### 6-2. インポート処理の段階

#### 1. アップロード

- インポートファイルを保存
- `import_files` を作成

#### 2. 自動判別

- ファイル種別
- 文字コード
- シート名 / ファイル名
- アグリノート出力らしさ

#### 3. 正規化

アグリノートの複数ファイルを、本アプリの中間形式に変換します。

例:

- 作業記録 一般 + 作業者 + 農薬 + 肥料 + 資材
  - `work_logs`
  - `work_log_targets`
  - `work_material_usages`

#### 4. マッピング確認

ユーザーに以下を見せるべきです。

- 圃場 -> 生産エリア の対応
- 作付 -> 作付サイクル の対応
- 品質・規格 -> 等級/規格の対応
- 単位の変換結果
- スキップ件数

#### 5. プレビュー

- 作成件数
- 更新件数
- 未解決件数
- 警告件数

#### 6. 反映

- `import_rows` に行単位の結果を保持
- 成功/失敗/警告を保存

---

### 6-3. 中間マッピングで必要な考え方

#### 圃場 -> 生産エリア

アグリノートは `圃場` ですが、本アプリは `生産エリア` です。基本は1対1対応で良いですが、将来的に次も必要です。

- 圃場1件 -> ハウス
- 圃場1件 -> 露地圃場
- 圃場1件 -> 親エリア

#### 作付 -> 作付サイクル

重要です。記録だけ入れても、作付サイクルが先にないと後続の収穫・出荷・分析に繋がりません。

したがって、推奨順は:

1. 圃場情報
2. 作付情報
3. 記録情報

#### IDの扱い

アグリノート出力には `作業ID` `作付ID` `圃場ID` などが含まれます。これを本アプリの内部IDにそのまま使うべきではありませんが、外部参照キーとして保持する価値があります。

推奨:

- `external_source = agrinote`
- `external_record_type`
- `external_record_id`

---

### 6-4. MVPとして最初に対応すべきインポート

#### 先に対応するもの

- 圃場情報
- 作付情報
- 作業記録 一般
- 作業記録 農薬 / 肥料 / 資材
- 収穫記録
- 出荷記録

#### 後回しでもよいもの

- 生育イベント
- 機械
- 作業者詳細
- 土地・不動産情報

理由:

- まずは「過去記録を捨てずに使い始められる」状態が重要
- 原価分析や人件費分析は第2段階でも良い

## 7. 開発順序の提案

### Phase 1

- `import_files` `import_rows` の実装
- アグリノート `圃場情報` `作付情報` の取込

### Phase 2

- 作業記録のアグリノート取込
- 収穫・出荷の取込
- 外部ID保持
- 取込プレビューUI

### Phase 3

- 生育記録/イベント取込
- 面積按分/単位変換の高度化

---

## 8. 最終提案

本アプリでは、`乗り換え障壁を下げること` が重要です。そのため、アグリノートからの `圃場 / 作付 / 作業記録 / 収穫 / 出荷` を段階的に受けられるようにする価値は大きいです。

最初に着手すべきは次です。

1. `import_files` `import_rows` の骨組み
2. アグリノートの圃場/作付インポート
3. 作業記録インポート

この順なら、導入時の移行負担を先に下げつつ、後続の収穫・出荷・分析にも繋げやすくなります。
